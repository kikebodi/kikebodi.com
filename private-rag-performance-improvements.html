Part 7: Performance improvements

At this point, your private RAG is already functional and measurable. Now we do what every production system needs: reduce latency, improve answer quality, and control cost without breaking reliability.

7.1 Profile first, optimize second

Before changing models or parameters, measure where time goes. Typical hotspots are:

  • Embedding latency during ingestion
  • Retriever and reranker latency at query time
  • Generation latency (especially with large local models)
import time


def timed(label: str, fn, *args, **kwargs):
    start = time.perf_counter()
    result = fn(*args, **kwargs)
    elapsed_ms = (time.perf_counter() - start) * 1000
    print(f"{label}: {elapsed_ms:.1f} ms")
    return result

7.2 Tune chunking and retrieval together

The biggest quality gains usually come from chunking + retrieval tuning. Treat them as a pair, not as independent knobs.

  • Start with chunk size 400–800 and overlap 10–20%
  • Test top-k values (e.g. 3, 5, 8, 10)
  • Prefer MMR when documents are repetitive
retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 6,
        "fetch_k": 24,
        "lambda_mult": 0.35,
    },
)

7.3 Add reranking only when it pays off

Rerankers improve precision, but they add latency. The usual winning setup is: retrieve 20-30 chunks, rerank to top 4-6, then send only those to the generation model. 

candidate_docs = retriever.invoke(question)

# pseudo-code for reranking
reranked = reranker.rank(question=question, documents=candidate_docs)
final_context = [item.document for item in reranked[:5]]

7.4 Cache aggressively at safe boundaries

Low-risk caches that usually provide immediate value:

  • Embedding cache (document fingerprint -> vector)
  • Retrieval cache (normalized query -> doc ids)
  • Response cache for repeated internal FAQs
import hashlib


def normalized_query_key(query: str) -> str:
    normalized = " ".join(query.lower().split())
    return hashlib.sha256(normalized.encode("utf-8")).hexdigest()

7.5 Add quality guardrails

Performance is not only speed. In production, you should fail safely when retrieval confidence is low.

  • Return "I don't know" when no chunk passes a minimum relevance threshold
  • Always cite source chunks in the final answer
  • Log low-confidence queries for offline tuning

7.6 Final checklist

Before considering your RAG production-ready, verify:

  • P95 latency and throughput under realistic load
  • Stable retrieval metrics (MRR, nDCG, keyword coverage)
  • No major regression in answer quality metrics
  • Cost per query aligned with business constraints

If you have reached this point, you now have a complete end-to-end private RAG workflow: ingestion, retrieval, generation, evaluation, and optimization.
